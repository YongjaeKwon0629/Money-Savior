# 가계부 기반 저축 운영 추천 서비스 MVP 개발 태스크 분해

## 1. 문서 개요

본 문서는 MVP 구현을 위한 개발 작업을 기능 단위로 분해하고 우선순위를 정의하기 위한 문서이다.  
목표는 기획 문서와 개발준비문서를 실제 개발 가능한 작업 단위로 전환하는 것이다.

## 2. 개발 우선순위 원칙

- 사용자 입력부터 결과 화면까지 한 번의 완결된 흐름을 먼저 만든다.
- 백엔드 계산 및 추천 로직은 MVP 규칙을 우선 그대로 구현한다.
- 저장 구조와 결과 조회 구조를 먼저 고정하고, 이후 UI 디테일을 보완한다.
- 부가 기능보다 핵심 흐름 완성을 우선한다.
- MVP 1차에서는 필수 API 2개와 핵심 화면 5개만으로도 동작 가능해야 한다.

## 3. MVP 핵심 개발 흐름

1. 사용자가 입력 화면에서 데이터를 입력한다.
2. 입력값이 서버로 전송된다.
3. 서버가 계산 및 추천 로직을 수행한다.
4. 추천 결과가 저장된다.
5. 결과 화면이 렌더링된다.

## 5. MVP 1차 구현 범위

MVP 1차에서는 아래 범위를 우선 구현한다.

### 백엔드

- DB 스키마
- Request validation
- FinanceCalculator
- RecommendationEngine
- ExceptionPolicyEngine
- POST /financial-inputs
- GET /recommendations/{inputId}

### 프론트

- 온보딩 화면
- 기본 정보 입력 화면
- 지출 정보 입력 화면
- 목표 및 성향 입력 화면
- 결과 화면

즉, 입력 수정 API와 입력 재조회 API는 1차 필수 범위가 아니다.

## 7. MVP 완료 기준

다음 조건을 만족하면 MVP 1차 구현 완료로 본다.

- 사용자가 입력 화면을 끝까지 완료할 수 있다.
- 입력값이 서버에 저장된다.
- 추천 결과가 계산 및 저장된다.
- 결과 화면에 권장 저축액과 추천 구조가 노출된다.
- 예외 상황에서 보수적 메시지가 정상 출력된다.
- 새로고침 또는 재조회 시 결과를 다시 가져올 수 있다.

## 9. 결론

MVP 개발 태스크 분해의 핵심은 사용자 입력부터 추천 결과 노출까지의 완결된 흐름을 가장 먼저 구현하는 것이다.  
특히 현재 구조에서는 백엔드 계산/추천 엔진과 프론트 입력/결과 화면을 명확히 나눠 구현하는 것이 가장 효율적이며, 이후 고도화 기능은 이 기본 흐름 위에 확장하는 방식이 적절하다.
