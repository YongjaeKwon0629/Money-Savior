# 가계부 기반 저축 운영 추천 서비스 MVP 백엔드 설계 초안

## 1. 문서 개요

본 문서는 가계부 기반 저축 운영 추천 서비스 MVP 구현을 위한 백엔드 구조와 책임 범위를 정의하기 위한 문서이다.  
본 문서의 목적은 API, 계산 로직, 추천 로직, 예외 처리, 데이터 저장 흐름을 명확히 분리하고, 실제 구현 시 백엔드 구조가 흔들리지 않도록 기준을 제공하는 데 있다.

## 2. 백엔드 목표

MVP 백엔드의 핵심 목표는 다음과 같다.

- 사용자 입력값을 저장한다.
- 입력값을 바탕으로 재무 계산을 수행한다.
- 계산 결과를 바탕으로 추천 방식을 결정한다.
- 추천 결과를 저장하고 결과 화면용 응답으로 조합한다.
- 예외 상황을 일관된 방식으로 처리한다.

## 3. 백엔드 책임 범위

MVP 백엔드는 아래 책임을 가진다.

- 입력값 유효성 검증
- 월별 입력값 저장
- 월 잉여자금 및 비상금 관련 계산
- 추천 방식 분기
- 목표 기반 추천 보정
- 예외 코드 판별 및 주의 메시지 생성
- 결과 화면용 응답 데이터 조합

MVP 백엔드는 아래 책임은 가지지 않는다.

- 금융상품 상세 비교
- 실시간 시세 조회
- 투자 자동 실행
- AI 코칭 결과 생성

## 4. 권장 백엔드 레이어 구조

MVP 백엔드는 아래 레이어 구조를 권장한다.

```text
Controller
  -> Service
    -> Domain Logic
      -> Repository
```

### 4.1 Controller

역할:

- HTTP 요청 수신
- Request validation 호출
- Service 호출
- 응답 반환

예시:

- `FinancialInputController`
- `RecommendationController`

### 4.2 Service

역할:

- 입력 저장과 추천 생성 흐름 오케스트레이션
- 여러 도메인 로직 조합
- 트랜잭션 경계 관리

예시:

- `FinancialInputService`
- `RecommendationService`

### 4.3 Domain Logic

역할:

- 순수 계산 로직 처리
- 추천 분기 처리
- 예외 정책 처리
- 응답 조합을 위한 중간 결과 생성

예시:

- `FinanceCalculator`
- `RecommendationEngine`
- `ExceptionPolicyEngine`
- `RecommendationActionFactory`

### 4.4 Repository

역할:

- DB 읽기/쓰기 처리
- 엔티티 영속화

예시:

- `UserRepository`
- `MonthlyFinanceInputRepository`
- `RecommendationResultRepository`

## 5. 권장 모듈 구조

## 6. 핵심 백엔드 흐름

### 6.1 추천 생성 흐름

`POST /financial-inputs`

1. Controller가 요청을 받는다.
2. Request DTO validation 수행
3. Service가 입력값 저장 흐름 시작
4. FinanceCalculator가 계산 수행
5. RecommendationEngine이 추천 방식 결정
6. ExceptionPolicyEngine이 예외 코드 및 주의 포인트 생성
7. RecommendationResult 저장
8. RecommendationActionFactory가 결과 화면용 actions 조합
9. Controller가 결과 응답 반환

### 6.2 결과 조회 흐름

`GET /recommendations/{inputId}`

1. Controller가 요청을 받는다.
2. Service가 입력 및 추천 결과 조회
3. RecommendationActionFactory가 actions 조합
4. 결과 화면용 DTO 반환

## 7. 핵심 백엔드 구성 요소

### 7.1 FinanceCalculator

역할:

- 월 잉여자금 계산
- 고정비 비중 계산
- 생활비 기준액 계산
- 비상금 목표액 계산
- 비상금 충족률 계산
- 안전/권장/도전 저축액 계산

입력:

- 월 실수령액
- 월 고정비
- 월 변동지출
- 비상금 보유액

출력:

- 계산 결과 객체

### 7.2 RecommendationEngine

역할:

- 저축 여력 단계 판단
- 비상금 상태 판단
- 저축 성향 반영
- 추천 방식 유형 결정
- 금액 배분 결정

입력:

- 계산 결과 객체
- 저축 목표
- 저축 성향

출력:

- 추천 결과 객체

### 7.3 ExceptionPolicyEngine

역할:

- 예외 코드 판별
- 경고 여부 판단
- 주의 메시지 생성
- 추천 방식 보수적 조정

입력:

- 입력값
- 계산 결과
- 추천 결과

출력:

- 예외 코드
- 주의 메시지 리스트
- 보정된 추천 결과

### 7.4 RecommendationActionFactory

역할:

- 추천 결과와 예외 상태를 기반으로 화면 액션 구성

예시 규칙:

- `EXPENSE_CONTROL` -> `EDIT_INPUT`, `RETRY_DIAGNOSIS`
- 일반 추천 결과 -> `SAVE_PLAN`, `RETRY_DIAGNOSIS`

입력:

- 추천 결과 유형
- 예외 코드

출력:

- 액션 리스트

## 8. DTO 설계 방향

### 8.1 Request DTO

- `CreateFinancialInputRequest`
- `UpdateFinancialInputRequest`

### 8.2 Response DTO

- `RecommendationResponse`
- `FinancialInputResponse`

### 8.3 내부 결과 DTO

- `CalculationResult`
- `RecommendationDecision`
- `ExceptionDecision`

## 9. 트랜잭션 처리 기준

`POST /financial-inputs`와 `PUT /financial-inputs/{inputId}`는 하나의 트랜잭션으로 처리하는 것을 권장한다.

이유:

- 입력 저장과 추천 결과 저장이 함께 일어나야 한다.
- 중간 저장만 되고 결과 저장이 실패하는 상황을 방지해야 한다.

## 10. 예외 처리 전략

### 10.1 유효성 예외

- 필수값 누락
- 숫자 범위 오류
- enum 값 오류

처리:

- 400 Bad Request 반환

### 10.2 도메인 예외

- 입력값으로 계산 불가능한 상태
- 추천 로직 충돌

처리:

- 가능하면 보수적 추천으로 전환
- 불가 시 500 또는 도메인 에러 응답

### 10.3 데이터 예외

- 존재하지 않는 inputId

처리:

- 404 Not Found 반환

## 11. AI 연동 위치

MVP에서는 AI를 백엔드 핵심 로직에 포함하지 않는다.

향후 확장 시에는 아래 위치에 붙이는 것을 권장한다.

- `AIExplanationService`
- `AICoachingService`

권장 구조:

```text
RecommendationService
  -> FinanceCalculator
  -> RecommendationEngine
  -> ExceptionPolicyEngine
  -> AIExplanationService (추후)
```

즉, AI는 계산 엔진 이후의 설명 레이어에 위치해야 한다.

## 12. 구현 우선순위

1. Request validation 구현
2. FinanceCalculator 구현
3. RecommendationEngine 구현
4. ExceptionPolicyEngine 구현
5. RecommendationActionFactory 구현
6. Repository 및 DB 저장 구현
7. Controller/API 연결

## 13. 결론

MVP 백엔드의 핵심은 입력값 저장, 계산, 추천, 결과 조합을 명확한 책임 단위로 분리하는 것이다.  
특히 계산 엔진과 추천 엔진, 예외 정책 엔진을 분리해두면 이후 AI 코칭 기능이나 추천 고도화 기능을 추가하더라도 구조를 크게 바꾸지 않고 확장할 수 있다.
