# 가계부 기반 저축 운영 추천 서비스 MVP 화면별 데이터 입출력 정의

## 1. 문서 개요

본 문서는 MVP 각 화면이 어떤 데이터를 입력받고, 어떤 데이터를 표시하며, 어떤 API와 연결되는지를 정의하기 위한 문서이다.  
본 문서의 목적은 프론트엔드 구현 기준을 명확히 하고, 화면과 API 간 연결 관계를 정리하는 데 있다.

## 2. 화면 목록

- 온보딩 화면
- 기본 정보 입력 화면
- 지출 정보 입력 화면
- 목표 및 성향 입력 화면
- 결과 화면

## 3. 화면별 데이터 정의

### 3.1 온보딩 화면

#### 목적

- 서비스 가치 전달 및 입력 시작 유도

#### 입력 데이터

- 없음

#### 출력 데이터

- 메인 헤드라인
- 서비스 설명 문구
- 핵심 가치 카드 문구

#### 액션

- `시작하기` 클릭 시 기본 정보 입력 화면으로 이동

#### API 연동

- 없음

### 3.2 기본 정보 입력 화면

#### 목적

- 기본 금융 정보 수집

#### 입력 데이터

사용자 직접 입력:

- `monthlyIncome`
- `paydayDay`
- `emergencyFundAmount`

#### 출력 데이터

- 필드 라벨
- 도움말 문구
- 진행 상태 값

#### 로컬 상태

```ts
{
  monthlyIncome: number | null,
  paydayDay: number | null,
  emergencyFundAmount: number | null
}
```

#### 검증 규칙

- `monthlyIncome > 0`
- `paydayDay >= 1 && paydayDay <= 31`
- `emergencyFundAmount >= 0`

#### 다음 화면 전달 데이터

- `monthlyIncome`
- `paydayDay`
- `emergencyFundAmount`

#### API 연동

- 없음

### 3.3 지출 정보 입력 화면

#### 목적

- 월간 지출 규모 수집

#### 입력 데이터

사용자 직접 입력:

- `fixedExpense`
- `variableExpense`

#### 출력 데이터

- 필드 라벨
- 고정비/변동지출 예시 문구
- 진행 상태 값

#### 로컬 상태

```ts
{
  fixedExpense: number | null,
  variableExpense: number | null
}
```

#### 검증 규칙

- `fixedExpense >= 0`
- `variableExpense >= 0`

#### 다음 화면 전달 데이터

- `fixedExpense`
- `variableExpense`

#### API 연동

- 없음

### 3.4 목표 및 성향 입력 화면

#### 목적

- 추천 분기 기준 수집

#### 입력 데이터

사용자 직접 입력:

- `savingGoal`
- `savingPreference`

#### 출력 데이터

- 목표 선택 항목
- 성향 선택 항목
- 진행 상태 값

#### 로컬 상태

```ts
{
  savingGoal: "EMERGENCY_FUND" | "LIFE_STABILITY" | "TRAVEL" | "HOUSING_MARRIAGE" | "INVESTMENT_PREP" | null,
  savingPreference: "STABLE" | "BALANCED" | "AGGRESSIVE" | null
}
```

#### 검증 규칙

- `savingGoal` 필수
- `savingPreference` 필수

#### 제출 시 서버로 전달할 전체 데이터

```json
{
  "targetMonth": "2026-04",
  "monthlyIncome": 2500000,
  "paydayDay": 25,
  "fixedExpense": 900000,
  "variableExpense": 1100000,
  "emergencyFundAmount": 300000,
  "savingGoal": "EMERGENCY_FUND",
  "savingPreference": "BALANCED"
}
```

#### API 연동

- `POST /api/v1/financial-inputs`

#### 구현 메모

- MVP 1차에서는 이 화면 제출 후 바로 결과 화면으로 이동한다.
- 입력 수정 기능이 없으면 별도 조회 API 없이도 전체 흐름 구현이 가능하다.

### 3.5 결과 화면

#### 목적

- 추천 결과 요약 및 실행 유도

#### 입력 데이터

서버 응답 데이터:

- `inputId`
- `recommendationId`
- `targetMonth`
- `summary`
- `savingAmounts`
- `recommendation`
- `reasons`
- `cautions`
- `actions`

#### 출력 데이터

상태 요약 영역:

- `summary.monthlyIncome`
- `summary.fixedExpense`
- `summary.variableExpense`
- `summary.surplusAmount`
- `summary.emergencyFundStatus`

저축 가능액 영역:

- `savingAmounts.safe`
- `savingAmounts.recommended`
- `savingAmounts.challenge`

추천 운영 방식 영역:

- `recommendation.type`
- `recommendation.parkingAccountAmount`
- `recommendation.installmentSavingsAmount`
- `recommendation.isaAmount`
- `recommendation.investmentAmount`

추천 이유 영역:

- `reasons[]`

주의 포인트 영역:

- `cautions[]`

실행 액션 영역:

- `actions[]`

#### 화면 렌더링용 프론트 모델 예시

```ts
{
  inputId: number,
  recommendationId: number,
  targetMonth: string,
  summary: {
    monthlyIncome: number,
    fixedExpense: number,
    variableExpense: number,
    surplusAmount: number,
    emergencyFundStatus: "NONE" | "INSUFFICIENT" | "SUFFICIENT"
  },
  savingAmounts: {
    safe: number,
    recommended: number,
    challenge: number
  },
  recommendation: {
    type: "EXPENSE_CONTROL" | "LIQUIDITY_FIRST" | "STABLE_SAVING" | "BALANCED_SAVING" | "DIVERSIFIED_ALLOCATION",
    parkingAccountAmount: number,
    installmentSavingsAmount: number,
    isaAmount: number,
    investmentAmount: number
  },
  reasons: string[],
  cautions: string[],
  actions: Array<{
    type: string,
    label: string
  }>
}
```

#### API 연동

- 기본 진입: `POST /api/v1/financial-inputs` 응답 사용
- 재조회: `GET /api/v1/recommendations/{inputId}`

#### 구현 메모

- 결과 화면은 `POST /financial-inputs` 응답만으로 바로 렌더링 가능해야 한다.
- `actions`는 응답 조합값을 그대로 사용한다.
- `GET /recommendations/{inputId}`는 새로고침, 재방문, 공유 링크 대응 시 활용한다.

## 4. 공통 상태 정의

프론트는 아래 공통 상태를 고려해야 한다.

| 상태 | 설명 |
|---|---|
| idle | 초기 상태 |
| editing | 사용자가 입력 중인 상태 |
| validating | 입력값 검증 중 |
| submitting | 추천 요청 전송 중 |
| success | 결과 수신 완료 |
| error | 요청 실패 또는 입력 오류 |

## 5. 데이터 흐름 요약

1. 사용자가 온보딩 화면에서 입력 플로우를 시작한다.
2. 프론트는 기본 정보, 지출 정보, 목표/성향 정보를 로컬 상태로 누적 저장한다.
3. 최종 제출 시 전체 입력 데이터를 `POST /financial-inputs`로 전송한다.
4. 서버는 입력값 저장, 계산 수행, 추천 결과 저장을 처리한다.
5. 프론트는 응답받은 결과 데이터로 결과 화면을 즉시 렌더링한다.
6. 필요 시 `GET /recommendations/{inputId}`로 결과를 다시 조회한다.

## 6. MVP 1차 구현 범위 기준

MVP 1차 구현에서는 아래 화면/연동만 우선 지원해도 된다.

- 온보딩 화면
- 기본 정보 입력 화면
- 지출 정보 입력 화면
- 목표 및 성향 입력 화면
- 결과 화면
- `POST /financial-inputs`
- `GET /recommendations/{inputId}`

아래 항목은 후속 구현 범위로 둘 수 있다.

- 입력값 조회 화면
- 입력값 수정 화면
- `GET /financial-inputs/{inputId}`
- `PUT /financial-inputs/{inputId}`

## 7. 구현 메모

- 입력 화면 간 이동 시 상태 보존이 필요하다.
- 숫자 입력은 문자열로 받아도 제출 직전 정규화가 필요하다.
- 결과 화면은 API 응답 구조를 최대한 그대로 사용할 수 있게 설계하는 것이 구현상 유리하다.
- 화면 문구는 프론트 하드코딩보다 API 응답 기반 구성을 우선 고려한다.

## 8. 결론

화면별 데이터 입출력 정의의 핵심은 입력 흐름과 결과 렌더링 흐름을 명확히 분리하는 것이다.  
MVP에서는 각 화면이 복잡한 상태를 가지지 않도록 하고, 최종 입력 데이터를 한 번에 제출해 결과를 받아오는 구조가 가장 단순하고 안정적이다.
